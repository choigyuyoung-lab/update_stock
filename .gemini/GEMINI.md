# 프로젝트 개발 및 토큰 최적화 가이드라인

## 1. 환경 및 아키텍처 기본 규칙
- **런타임 환경:** 본 프로젝트는 GitHub Codespaces 및 로컬 환경의 **Python 3.10** 버전을 표준으로 삼는다. 3.10 버전에 존재하지 않는 상위 버전 전용 문법(예: 3.11의 튜닝된 Exception Notes 등)은 절대 사용하지 않는다.
- **웹 크롤링 표준:** 웹 데이터 수집 시 반드시 `requests`로 raw 데이터를 받아 `StringIO`로 감싼 뒤 `pandas.read_html`을 사용하는 구조를 기본으로 채택한다.
- **성능 최적화:** 대량 데이터 수집 및 배치 처리가 필요할 때는 `ThreadPoolExecutor`를 활용하여 연동 속도를 최적화한다.

## 2. 극도의 토큰 소모량(Rate Limit) 절감 규칙 [필수]
- **Diff 중심 코드 제안 (최우선):** 코드를 수정하거나 제안할 때 전체 파일 코드를 통째로 출력하지 마라. 반드시 수정이 필요한 '특정 함수'나 '변경된 부분(Diff)'만 조각내어 출력할 것.
- **불필요한 설명 및 데코레이션 금지:** 코드 변경 시 원인과 결과에 대한 장황한 줄글 설명을 생략하고, 즉시 적용 가능한 코드와 핵심 요약만 제공하라. 인사말, 맺음말 등 불필요한 토큰 소비 문구를 전면 금지한다.
- **주석 및 문서화 최소화:** 코드 내부에 과도한 Docstring이나 당연한 내용의 주석을 달아 파일 크기(입력 컨텍스트)를 키우지 마라. 코드는 최대한 self-explanatory하게 작성한다.
- **단계별 승인(Task Atomization):** AI 에이전트는 한 번에 여러 파일을 수정하는 계획을 세우지 않는다. 오직 하나의 기능, 하나의 파일 단위로 유저의 승인을 받으며 전진한다.

## 3. 에이전트 행동 및 권한 제어
- 작업 영역 외의 파일을 탐색하거나 수정하기 전에 반드시 유저에게 터미널이나 채팅창으로 명확한 실행 계획을 보고하고 승인을 얻어야 한다.
- 코드 수정 후에는 유저가 Git 커밋을 효율적으로 진행할 수 있도록 한 줄 짜리 직관적인 커밋 메시지 템플릿만 결과물 끝에 제공한다.
